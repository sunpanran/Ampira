# Reviewer notes

1. Installing the extension replaces `chrome://newtab` with `dashboard.html`.
2. The three-step onboarding explains the New Tab override and local bookmark handling, lets the user choose bookmark folders, then requests optional exact website-origin and Chrome-provided website-icon access before opening the dashboard. The detailed AI data-transfer disclosure and affirmative consent control appear in Settings → AI before an AI key can be saved, tested, or used.
3. Core dashboard use requires only the `bookmarks`, `storage`, and `alarms` permissions. The extension only calls bookmark read methods. The optional `favicon` permission is requested from onboarding or Settings → Browser and can be removed there; without it, Ampira uses its packaged fallback icon.
4. To test Feed fetching, grant the listed exact origins in the final onboarding step. Settings → News shows effective coverage and per-source diagnostics, including the extraction method and a single-source retry. If an allowed landing page declares a Feed on another domain, that exact domain appears as a separate user-gesture permission request and receives no request before approval. All origins remain individually manageable under Settings → Browser; declining leaves bookmark cards usable.
5. To test images without Brave, grant exact source origins under Settings → Browser and open a new tab. Inspiration cards read only inert public page metadata, try up to three original-page candidates, and start preloading all 15 fixed daily cards across the three reshuffle batches before first render. An authorized same-origin news article with no Feed image may receive one bounded metadata read; cross-origin article metadata reads are skipped. To test fallback, optionally enable Brave Image Search, grant `api.search.brave.com`, and use a non-production test key. A news or inspiration card with no usable original image, or with deliberately failed original candidates, may then use Brave; unrelated URLs and Reader images remain excluded. Revoking a source origin stops its metadata reads and removes the corresponding image cache while leaving the bookmark usable.
6. AI is optional. To test it, open Settings → AI, review and affirmatively accept the disclosure, grant the configured provider origin, enter a non-production test key, save, and run the connection test. The key is sent directly to that provider to authenticate the request and never to an Ampira backend. Remove the key afterward.
7. The first compact efficiency card cycles through Today's events → Weather forecast → To-do list with one persistent Switch button. To test Weather, switch once, enter `慈溪`, and approve only `https://geocoding-api.open-meteo.com/*` and `https://api.open-meteo.com/*`. The bundled GeoNames-derived China index returns `慈溪市 · 宁波市 · 浙江省 · 中国` without sending that search text to a geocoder; its attribution is visible below the candidates. A non-China city query falls back directly to Open-Meteo's geocoder. Ampira never auto-selects ambiguous candidates or uses geolocation. Only the selected coordinates go directly to Open-Meteo's forecast API, not to an Ampira backend. Settings → Browser lists both origins after opt-in; revoke either one and confirm requests stop, the forecast cache is removed, and the selected city remains available for reauthorization. Weather attribution is visible in the forecast card.
8. To test To-do, switch again, add an item with Return, complete, restore, and delete it. To-dos and the selected utility mode remain local to the current Chrome profile and do not use Chrome Sync.
9. The extension contains no content scripts, remote JavaScript, `eval`, tracking, advertising, payment flow, or developer backend.
10. The in-app reader fetches public HTML only after exact-origin permission, then renders inert structured text and images. Videos remain explicit source links; no remote iframe or player is embedded.
11. Incognito use is disabled. General web search is not provided; the top search field filters Ampira content only.
