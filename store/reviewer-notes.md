# Reviewer notes

1. Installing the extension replaces `chrome://newtab` with `dashboard.html`.
2. The four-step onboarding explains the New Tab override and local bookmark handling, lets the user choose packaged or personal sources, requests optional exact website-origin and Chrome-provided website-icon access, then offers an optional AI-configuration handoff before opening the dashboard. Its compact inspiration-folder menu fixes Ampira’s packaged preset as the first option and lists personal bookmark folders below it. Preset mode can finish with no top-level inspiration folder. Switching later preserves the personal folder and never writes bookmarks. The detailed AI data-transfer disclosure and affirmative consent control appear in Settings → AI before an AI key can be saved, tested, or used.
3. The required permissions are `activeTab`, `bookmarks`, `storage`, and `alarms`. `activeTab` is used only after the user clicks the Ampira toolbar icon to read that page's title and URL and save it to the reading queue; it does not monitor tabs, page contents, or browsing history. Reading-queue records remain local unless the user separately enables the default-off Chrome Sync switch. The extension only calls bookmark read methods. The optional `favicon` permission is requested from onboarding or Settings → Browser and can be removed there; without it, Ampira uses its packaged fallback icon. The optional `search` permission is requested only from Settings → Browser and uses Chrome's Search API with the user's current default provider; removing it restores Ampira content search in the top field.
4. To test Feed fetching, grant the listed exact origins in the third onboarding step. Settings → News shows effective coverage and per-source diagnostics, including the extraction method and a single-source retry. If an allowed landing page declares a Feed on another domain, that exact domain appears as a separate user-gesture permission request and receives no request before approval. All origins remain individually manageable under Settings → Browser; declining leaves bookmark cards usable.
5. In preset mode, verify the 48 links use 24 packaged local covers, each cover appears on exactly two sites, daily batches do not repeat a cover, and the first three reshuffles contain 15 different sites and covers. Preset covers make no metadata, remote-image, or Brave requests and the 48 origins are absent from the permission list. In personal-bookmark mode, grant exact source origins under Settings → Browser and open a new tab. Personal inspiration cards read only inert public page metadata, try up to three original-page candidates, and start preloading all three reshuffle batches before first render. An authorized same-origin news article with no Feed image may receive one bounded metadata read; cross-origin article metadata reads are skipped. To test backup image search, open Settings → AI → Backup image search, enable Web image search, grant `api.search.brave.com`, and use a non-production test key. A news or personal inspiration card with no usable original image, or with deliberately failed original candidates, may then use Brave; unrelated URLs, preset covers, and Reader images remain excluded. Revoking a source origin stops its metadata reads and removes the corresponding image cache while leaving the bookmark usable.
6. AI is optional. The initial public-source permission step already lists all nine exact domains in the current language pack, including three reserved cross-language sources; those three origins receive no Feed request until AI is fully configured. To test AI, open Settings → AI, review and affirmatively accept the disclosure, grant the configured provider origin, enter a non-production test key, save, and run the connection test. The reserved sources then become active automatically. Raw cross-language items remain absent from the Feed, fallback brief, local search, AI search context, and visible counts unless the localized title and summary pass the interface-language check. The key is sent directly to that provider to authenticate the request and never to an Ampira backend. Remove the key afterward.
7. The first card under Information overview cycles through Today's events → Weather forecast → To-do list with one persistent Switch button. To test Weather, switch once, enter `慈溪`, and approve only `https://geocoding-api.open-meteo.com/*` and `https://api.open-meteo.com/*`. The bundled GeoNames-derived China index returns `慈溪市 · 宁波市 · 浙江省 · 中国` without sending that search text to a geocoder; its attribution is visible below the candidates. A non-China city query falls back directly to Open-Meteo's geocoder. Ampira never auto-selects ambiguous candidates or uses geolocation. Only the selected coordinates go directly to Open-Meteo's forecast API, not to an Ampira backend. Settings → Browser lists both origins after opt-in; revoke either one and confirm requests stop, the forecast cache is removed, and the selected city remains available for reauthorization. Weather attribution is visible in the forecast card.
8. To test To-do, switch again, add an item with Return, complete, restore, and delete it. To-dos, the selected weather city, and reading queue remain local by default. In Settings → Browser → Cross-device content, each category has a separate default-off Chrome Sync switch. Enable one, confirm the corresponding content appears in another signed-in synced Chrome installation, then disable it and confirm the Chrome Sync copy is removed while the current local copy remains. The selected utility mode and forecast cache never sync.
9. The extension contains no content scripts, remote JavaScript, `eval`, tracking, advertising, payment flow, or developer backend.
10. The in-app reader fetches public HTML only after exact-origin permission, then renders inert structured text and images. Videos remain explicit source links; no remote iframe or player is embedded.
11. To test toolbar capture, open an HTTPS page and click the Ampira icon. A packaged popup immediately saves the page and shows the localized result plus captured title and host; a purple check badge provides secondary feedback. Clicking again reports that the item is already queued and keeps one entry. Unsupported Chrome pages show an amber warning state and `!` badge. Open a new tab and confirm the captured page appears in Reading queue. With Sync reading queue disabled it remains local; with that separate opt-in enabled it follows the same bounded Chrome Sync path as card-added items.
12. Incognito use is disabled. By default the top search field filters Ampira content only. To test optional browser search, open Settings → Browser, enable Top search, enter `Ampira browser search QA`, and confirm Chrome's current default provider replaces the requesting tab. The navigation AI/content-search entry remains unchanged. Disable Top search and confirm the original Ampira filtering and content-search behavior returns.
13. To test Factory reset, use only synthetic content and non-production credentials. Open Settings → About → Import & export, cancel once and confirm no data changes, then confirm the destructive action. Verify local and synchronized Ampira data are removed, optional website/favicon/search permissions are revoked, Chrome bookmarks remain unchanged, and the dashboard restarts at the four-step onboarding flow. Synced deletions affect other signed-in installations, but their local-only data is not remotely erased.
